# API Contract: echolet Firmware <-> Server

## Endpoint

`POST /api/v1/upload-audio`

Назначение: отправка аудиофайла с устройства `echolet` на сервер для сохранения и последующей обработки.

## Headers

Обязательный заголовок:

```http
Authorization: Bearer <TOKEN>
```

## Content-Type

```http
multipart/form-data
```

## Поля формы

### `file`

- Тип: WAV-файл
- Обязательное поле
- Содержит аудиоданные, записанные устройством

### `device_id`

- Тип: строка
- Обязательное поле
- Пример: `echolet-001`

### `created_at`

- Тип: строка
- Обязательное поле
- Формат: ISO timestamp
- Пример: `2026-05-29T10:30:45Z`

### `battery`

- Тип: строка или `null`/`unknown`
- Поле статуса батареи на момент записи
- Примеры:
  - `87`
  - `unknown`
  - `null`

### `firmware_version`

- Тип: строка
- Версия прошивки устройства
- Пример: `0.1.0`

## Успешный ответ

При успешном приёме сервер возвращает JSON:

```json
{
  "ok": true,
  "status": "received",
  "event_id": "echolet-20260529133045",
  "transcription_status": "pending",
  "whisper_model": "medium",
  "inbox_dir": "80-89 AI/85 Echolet"
}
```

### Значение полей ответа

- `ok`: признак успешного приёма
- `status`: текущий статус обработки на стороне сервера
- `event_id`: идентификатор server-side capture
- `transcription_status`: начальный статус транскрибации
- `whisper_model`: модель `faster-whisper`, с которой принят запрос
- `inbox_dir`: inbox-папка внутри Obsidian vault, куда ляжет итоговый markdown

## Ошибки

Сервер может вернуть ошибку одного из следующих типов:

- `unauthorized`
- `missing_file`
- `invalid_file`
- `storage_error`
- `processing_failed`

Смысл ошибок:

- `unauthorized`: отсутствует токен или токен неверный
- `missing_file`: поле `file` не передано
- `invalid_file`: файл передан, но не соответствует ожидаемому формату или повреждён
- `storage_error`: сервер не смог сохранить файл или metadata/event
- `processing_failed`: файл принят, но последующий шаг обработки завершился ошибкой

## Правило для Firmware

Firmware должна считать файл успешно отправленным только если сервер вернул:

```json
{
  "ok": true
}
```

Правило очереди:

- если сервер вернул `ok=true`, файл можно пометить как `sent`
- если сервер вернул ошибку или вообще не ответил, файл должен остаться в очереди

Это нужно, чтобы не терять записи при сбоях сети, auth или storage.

## Правило для Server

Обязательные правила для сервера:

- сервер не должен удалять файл после приёма
- сервер должен сохранить исходное аудио во временное storage
- сервер должен создать markdown-файл во входящей Obsidian inbox-папке
- сервер должен скопировать оригинальный аудиофайл в `_attachments`
- если транскрибация не удалась, сервер всё равно должен создать markdown с ошибкой

Это гарантирует, что:

- оригинальный аудиофайл остаётся доступным для повторной обработки
- downstream-обработка может быть добавлена позже без потери данных
- состояние ingest фиксируется независимо от наличия транскрипции
- OpenClaw может обработать входящий capture даже при сбое `faster-whisper`
